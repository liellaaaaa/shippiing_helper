## 2. 三源合并与冲突规则

### 2.1 合并逻辑（`ledger_service.py` → `merge_preview()`）

以 **PI合同表**为主数据源（产品级数据最完整），逐产品合并：

1. 从 PI合同表获取：internal_code、quantity_kg、total_amount、hs_code、customs_name、customs_ingredients、product_appearance
2. 从销售订单表按 internal_code 匹配补充：product_cn、spec_kg
3. 从 PI合同文件按 internal_code 匹配校验：quantity、total_amount（交叉验证，容差 ±0.01）
4. 未匹配字段回退到知识库（customs_name_service / customs_declaration_service）

### 2.2 字段优先级

| 字段 | 第一优先 | 第二优先 | 第三优先 |
|------|----------|----------|----------|
| hs_code | PI合同表 | PI合同文件 | 知识库 |
| customs_name | PI合同表 | 销售订单表 | 知识库 |
| customs_ingredients | PI合同表 | 知识库 | — |
| product_appearance | PI合同表 (product_color) | — | — |
| quantity_kg | PI合同表 | 销售订单表 | PI合同文件(校验) |
| unit_price | PI合同表 | PI合同文件 | 销售订单表 |
| total_amount | PI合同表 | PI合同文件 | 销售订单表 |
| consignee_name | PI合同文件 | — | — |
| consignee_address | PI合同文件 | — | — |
| destination | PI合同文件 | — | — |
| loading_port | PI合同文件 | — | — |
| price_term | PI合同文件 | — | — |
| payment_terms | PI合同表 | PI合同文件 | — |
| shipment_title | 销售订单表 / PI合同表 | — | —（两者应一致） |
| pi_no | PI合同表 | PI合同文件 | — |
| product_cn | 销售订单表 | — | — |
| spec_kg | 销售订单表 | — | — |
| merchandiser | 销售订单表 | — | — |
| order_date (交期) | 销售订单表 | — | — |

### 2.3 交叉校验

PI合同表 vs PI合同文件：
- **数量校验：** |quantity_kg - pf.quantity| > 0.01 → warning
- **金额校验：** |total_amount - pf.total_amount| > 0.01 → warning
- 校验结果在合并预览页面展示，用户确认后入库

## 4. 第二阶段：文档填充规则

### 4.1 订舱单 (Booking)

**API：** `POST /api/v1/documents/booking`
**模板：** xlsx，使用 `{{FIELD_NAME}}` 标记符
**填充方式：** 直接操作 xlsx zip 内部 XML 替换单元格值

| 标记符 | 字段名 | 数据来源 | 说明 |
|--------|--------|----------|------|
| `{{SHIPPER}}` | 发货人 | **shipment_title** 映射 | 见 4.1.1 |
| `{{CONSIGNEE}}` | 收货人 | **consignee_name + consignee_address** | 见 4.1.2 |
| `{{NOTIFY}}` | 通知方 | 与 CONSIGNEE 相同 | 通常与收货人一致 |
| `{{CUT_OFF_DATE}}` | 截关日期 | delivery_date | 销售订单表交货日期 |
| `{{PLACE_OF_RECEIPT}}` | 收货地 | **loading_port** | PI合同文件装货港 |
| `{{POL}}` | 装货港 | **固定选项** | 见 4.1.3 |
| `{{POD}}` | 目的港 | destination | PI合同文件目的港 |
| `{{PLACE_OF_DELIVERY}}` | 交货地 | destination | 与 POD 一致 |
| `{{MARKS}}` | 唛头 | "N/M" | 固定值 |
| `{{NO_KIND_PKG}}` | 件数 | pallet_count + " PALLETS" | 包装计算 |
| `{{DESC1}}`~`{{DESC6}}` | 货物描述 | customs_name（每产品一个） | PI合同表 |
| `{{GROSS_WEIGHT}}` | 毛重 | gross_weight_kg | 包装计算 |
| `{{MEASUREMENT}}` | 体积 | volume_cbm + " CBM" | 包装计算 |

#### 4.1.1 SHIPPER 映射规则

SHIPPER 来自台账的 `shipment_title`（出货抬头），需要映射为对应的英文公司名：

| shipment_title | SHIPPER 值 |
|----------------|-----------|
| 宏昊 | GUANGDONG HONGHAO CHEMICAL CO., LTD. |
| （其他抬头） | 需建立映射表，或从数据中动态获取 |

**实现要求：** 创建 `shipment_title_mapping` 配置或 JSON，维护中文抬头 → 英文 SHIPPER 的映射关系。SHIPPER 不能 hardcode。

#### 4.1.2 CONSIGNEE 拼接规则

CONSIGNEE 需要拼接为完整的收货人信息块：

```
{consignee_name}
{consignee_address}
{电话（如有）}
```

实际数据样例：
```
TOA-DOVECHEM INDUSTRIES CO., LTD
31/5 MOO 3,BANGNA TRAD RD.KM23,BANGSAOTHONG,
AMPHUR BANGSAOTHONG,SAMUTPRAKARN 10570,THAILAND.
TEL:+66 2 7400822 TO 26
```

**注意：** 电话号码目前未从 PI 合同文件中单独提取。需确认 PI 文件中是否有 TEL 字段需要补充提取。如果有，需在 `pi_parser.py` 的 `parse_proforma_invoice()` 中增加 TEL 提取逻辑。

#### 4.1.3 POL（装货港）规则

POL 为固定选项，不从数据源提取：

| 选项 | 说明 |
|------|------|
| NanSha, China | 南沙港（默认） |
| SheKou, China | 蛇口港 |

**实现方式：** 前端提供下拉选择，默认值为 "NanSha, China"。或根据 loading_port 的值自动判断（如 loading_port 包含 "Nansha" 则选南沙，"Shekou" 则选蛇口）。

### 4.2 LOI 保函

**API：** `GET /api/v1/documents/loi?order_no=xxx&pi_no=xxx`
**模板：** docx，使用 `{{placeholder}}` 占位符
**填充方式：** python-docx 段落文本替换

**LOI 保函仅需填充 5 个业务字段 + 日期，共 6 个值：**

| 占位符 | 字段 | 数据来源 | 说明 |
|--------|------|----------|------|
| `{{product_name_cn}}` | 中文品名 | **运输鉴定报告** product_name_cn | 通过产品名匹配运输鉴定报告 |
| `{{product_name_en}}` | 英文品名 | **运输鉴定报告** product_name_en | 报告中 "Name of Goods" 行提取 |
| `{{report_no}}` | 鉴定书编号 | **运输鉴定报告** report_no | 报告中 "No." 前缀提取 |
| `{{sample_desc}}` | 产品外观与性状 | **运输鉴定报告** sample_desc_cn | 报告中"样品描述"段落提取 |
| `{{product_usage}}` | 产品主要用途 | **申报要素.json** "用途" 字段 | 通过 hs_code 查询 |
| `{{date}}` | 日期 | 当前日期 | 格式："YYYY 年 MM 月 DD 日" |

**数据源变更说明：** LOI 的数据不再直接从台账取字段，而是：
1. 通过产品中文名在运输鉴定报告索引（`transport_reports` 表）中模糊匹配
2. 从匹配到的报告 PDF 中提取前 4 个字段（report_no、product_name_en、sample_desc_cn）
3. 通过台账中的 hs_code 查 `申报要素.json` 获取第 5 个字段（用途）
4. 日期取当前时间

**运输鉴定报告提取逻辑已实现：** `transport_service.py` → `extract_fields()` 方法可提取：
- report_no（报告编号）
- product_name_en（英文品名，从 "Name of Goods" 行）
- sample_desc_cn（中文样品描述）
- sample_desc_en（英文样品描述）
- product_name_cn（中文品名，通过英文名查品名映射表反推）

**shipper/consignee/port 等字段不需要填入 LOI。**

### 4.3 MSDS 安全数据表

**API：** `GET /api/v1/documents/msds?product=xxx`
**模板：** docx（从 MSDS 目录匹配或标准模板）
**填充方式：** python-docx 表格单元格填充

| 填充位置 | 字段 | 数据来源 |
|----------|------|----------|
| 表1 第1行第2列 | 外观与性状 (physical_form) | MSDS 文档提取 |
| 表1 第2行第2列 | 离子型 (ion_type) | MSDS 文档提取 |
| 表1 第3行第2列 | pH值 (ph) | MSDS 文档提取 |
| 表2 各行 | 成分组成 (component/cas/percentage) | MSDS 文档提取 |

**流程：** 按产品中文名在 `msds_indexes` 表中模糊匹配 → 找到对应 MSDS 文件 → 提取文本 → 解析成分表和理化特征 → 填充模板。
